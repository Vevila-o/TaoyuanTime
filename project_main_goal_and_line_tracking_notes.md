# 專題主要目標與 LINE 互動追蹤備忘錄

日期：2026-05-27

## 1. 專題主軸

本專題提出「AI 智慧公共資訊整合與個人化通知系統」的概念，透過資料爬蟲、自動分類與資訊通知等技術，建立一套能夠整合不同來源公告資訊的系統。

系統可自動蒐集政府機關、學校或公共單位所發布的公告內容，並透過 AI 分析產生摘要與分類標籤，使資訊更容易被整理與搜尋。此外，系統也可結合通知與行事曆整合功能，使使用者能即時掌握重要活動或截止日期。

透過此系統，希望能提升公共資訊的可取得性與使用效率，並為未來智慧城市中的資訊服務模式提供新的應用方向。

目前專題落地場景以「桃園公共活動資訊」為主要示範領域，使用 LINE Bot 作為前台使用者入口，Django + SQLite 作為後端與資料儲存基礎。

## 2. LINE 卡片點擊追蹤策略

LINE 卡片互動設計要依目的選擇不同 action 類型。

### 2.1 只想記錄誰點了卡片

使用 `postback`。

適用情境：

- 使用者點「有興趣」
- 使用者點「沒興趣」
- 使用者點「訂閱」
- 使用者點「查看更多」
- 使用者點「加入行事曆」前先進入系統流程
- 使用者點「怎麼去」但仍由 Bot 回覆交通資訊

優點：

- LINE 會把 postback event 傳回 webhook。
- 後端能知道是哪個 `line_user_id` 點了哪個 action。
- 可以直接寫入 `ActionLog`。
- 適合推薦權重、使用者偏好、互動分析。

資料庫可記錄：

- user
- activity
- action_type
- metadata
- created_at

例如：

```text
action_type = view_card / interested / not_interested / subscribe / how_to_go / add_calendar / view_more
```

### 2.2 點一下直接進活動網站，又要記錄

使用 `uri`，但不要直接放官方活動網址。

應改放系統自己的中轉 tracking URL。

流程：

```text
LINE 卡片 URI
  -> https://你的網域/track/activity/<activity_id>?user=<token>&action=view_detail
  -> 後端記錄 ActionLog
  -> 302 redirect 到 official_detail_url / source_url
```

用途：

- 使用者點「查看詳情」後，仍能直接進入桃園觀光導覽網或官方活動頁。
- 系統同時能記錄使用者點擊。
- 後台可統計各活動點擊數。

注意事項：

- URL 不應直接暴露 LINE user id。
- 建議使用短 token 或 server-side mapping。
- tracking endpoint 應驗證 activity 是否存在。
- redirect 目的地應只允許官方或可信來源，避免 open redirect 風險。

## 3. 前台基本功能

### 3.1 使用者註冊與偏好設定

使用者首次進入 LINE Bot 後，系統引導其完成基本設定流程。

設定內容包含：

- 感興趣的活動類型標籤，例如節慶、親子、展覽、市集。
- 所在或偏好行政區域，例如中壢區、桃園區、大溪區。

設定完成後，系統建立個人化檔案，作為後續活動推薦與推播通知依據。

使用者可隨時於選單中進入「偏好設定」重新調整標籤內容。

對應資料表：

- `UserProfile`
- `Tag`
- `UserProfile.preferred_tags`

### 3.2 地區標籤篩選

使用者可設定一個或多個偏好地區標籤。

系統依據所選地區自動過濾活動資訊，確保推播與瀏覽結果只顯示相關地區活動。

例如：

```text
使用者設定偏好：中壢區
系統推薦結果：優先或僅推薦中壢區活動
```

設計目的：

- 避免不相關地區資訊干擾。
- 提升資訊精準度。
- 改善使用體驗。

### 3.3 活動主動查詢與網頁跳轉

使用者可透過 LINE Bot 快捷選單主動搜尋活動。

當使用者想取得完整活動資訊時，可點選活動卡片中的「查看詳情」按鈕。

設計要求：

- 使用者最後要能跳轉到官方活動頁面。
- 系統若要記錄點擊，需使用中轉 tracking URL。
- 進入官方頁後，使用者可確認報名方式、活動規則、最新公告。

### 3.4 活動卡片與市民卡優惠提示

使用者可透過 LINE Bot 快捷選單瀏覽近期活動。

系統預設依時間排序顯示即將舉辦的活動卡片。

每張卡片包含：

- 活動名稱
- 時間
- 地點
- 費用類型
- 簡短摘要

如果系統偵測到活動有桃園市民卡相關優惠，活動提醒推播時應同步顯示市民卡優惠提示。

可延伸功能：

- 提供 QR Code 給使用者現場出示核驗。
- 後台標記活動是否有市民卡優惠。

### 3.5 交通時刻查詢：How to go

使用者可於活動卡片中點選「怎麼去」按鈕。

完整目標：

- 根據活動地點與使用者所在位置。
- 整合台鐵、捷運、市區公車、活動接駁車等交通資訊。
- 提供即時大眾運輸建議與時刻表。

POC 階段可先提供：

- 活動地點
- Google Maps 連結
- 基本交通提示

互動紀錄：

- 使用者是否點擊「怎麼去」應寫入 `ActionLog`。
- 可作為活動參與意願與推薦權重調整依據。

### 3.6 AI 對話框與活動推廣提示

使用者可於 LINE Bot 對話框中以自然語言查詢活動資訊。

AI 除回覆查詢結果，也可在對話過程中主動提及與使用者興趣相關的活動優惠或推廣資訊。

例如：

- 限時優惠
- 早鳥報名
- 市民卡專屬折扣

限制：

- AI 回覆必須依據資料庫中的活動資料。
- AI 不得自行編造活動。

### 3.7 訂閱管理與推播通知

使用者可針對感興趣活動點選「訂閱」。

系統應記錄訂閱紀錄，並於下列時間自動提醒：

- 活動開放報名前
- 活動開始前一週

使用者可於個人頁面統一管理訂閱清單，並自行調整：

- 推播頻率
- 通知類型

目的：

- 避免資訊過載。
- 提升活動提醒實用性。

### 3.8 行事曆整合

使用者可於活動卡片中點選「加入行事曆」。

系統將活動資訊新增至使用者個人行事曆。

活動資訊包含：

- 活動名稱
- 活動時間
- 活動地點
- 官方詳情 URL
- 活動摘要

POC 可先產生 Google Calendar URL 或 `.ics` 檔案。

## 4. 後台基本功能

### 4.1 活動資料管理

系統管理者可於後台管理介面檢視已匯入活動資料。

應顯示欄位：

- 活動名稱
- 時間
- 地點
- 費用
- 分類標籤
- 來源網址
- 狀態
- 資料品質

管理者可操作：

- 新增
- 編輯
- 下架
- 刪除

目前狀態：

- 活動列表已改為讀 `Database.events.Activity`。
- 新增、編輯、下架按鈕尚未完整接正式資料表。

### 4.2 AI 活動標籤審核

後台應提供 AI 自動分類標籤審核機制。

流程：

1. 系統或 AI 根據活動內容產生候選標籤。
2. 候選標籤只能來自固定 Tag 池。
3. 管理者逐筆確認標籤是否正確。
4. 管理者可修正或補充。
5. 確認後才影響搜尋與推薦。

目的：

- 確保 Tag 池一致性。
- 避免錯誤標籤影響推薦結果。

目前缺口：

- 尚未有 `TagReview` 或 `ActivityTagSuggestion` 類資料表。

### 4.3 數據儀表板與使用分析

後台應提供視覺化儀表板。

應包含：

- 活動數據：
  - 各活動點擊次數
  - 訂閱人數
  - 行事曆加入次數

- 標籤數據：
  - 活動類型 Tag 熱門排行
  - 地區 Tag 熱門排行

- 使用者互動數據：
  - 使用者活躍度
  - 推播開啟率
  - 沒興趣回饋比例

- 假日與時段分析：
  - 活動集中日期
  - 假日活動分布
  - 熱門活動時段

目前缺口：

- `ActionLog` 可支援部分互動分析。
- 但尚缺推播 campaign、delivery log、calendar log 等更完整資料。

### 4.4 推播管理

管理者可於後台手動設定特定活動的推播：

- 推播對象
- 推播時間
- 推播內容
- 對應活動

用途：

- 重大節慶
- 政府重點推廣活動
- 補充 AI 自動推播無法涵蓋的特殊情境

目前缺口：

- 尚缺 `PushCampaign`
- 尚缺 `PushDeliveryLog`

## 5. AI 系統設計

### 5.1 AI 活動摘要與 OCR 圖片資訊擷取

桃園觀光導覽網部分活動頁面可能只有簡短文字說明，而完整資訊可能在活動海報圖片中。

因此系統預計導入具備 OCR 與多模態分析能力的 AI API。

AI 要協助擷取：

- 活動時間
- 活動地點
- 費用
- 報名方式
- 注意事項
- 市民卡優惠
- 圖片文字

AI 整理後應產生：

- LINE 可讀的簡短摘要
- 活動詳情摘要
- 結構化欄位

成本控制：

- OCR 擷取後文字與摘要要存入 SQLite。
- 後續 LINE 推播、活動卡片、AI 搜尋直接使用資料庫內容。
- 避免同一活動重複呼叫 AI API。

### 5.2 固定 Tag 池與活動分類

為避免 AI 自行產生過多不一致或難以管理的標籤，系統採固定 Tag 池。

AI 僅能依照既有 Tag 進行判斷與搜尋，不得自行新增未審核標籤。

Tag 分四類：

1. 地區 Tag
   - 桃園區
   - 中壢區
   - 八德區
   - 大溪區
   - 龍潭區
   - 龜山區

2. 活動類型 Tag
   - 節慶
   - 市集
   - 展覽
   - 親子
   - 藝文
   - 戶外
   - 導覽
   - 體驗
   - 講座
   - 音樂
   - 表演

3. 費用 Tag
   - 免費
   - 付費
   - 需報名
   - 免報名
   - 金額未提供

4. 優惠 Tag
   - 桃園市民卡優惠
   - 附近市民卡合作店家
   - 無市民卡資訊
   - 優惠未提供

目前資料庫已有固定 Tag 池，但仍需補齊命名一致性，例如「桃園市民卡優惠」與目前 seed 中的「市民卡」需要對齊。

### 5.3 AI 自然語言搜尋

使用者可在 LINE 對話框用自然語言查詢活動。

範例：

- 這週末桃園有什麼活動？
- 中壢最近有活動嗎？
- 有沒有親子活動？
- 免費活動有哪些？

流程：

1. 使用者輸入自然語言。
2. AI 判斷查詢條件。
3. 條件包含：
   - 時間
   - 地區
   - 活動類型
   - 費用
   - 關鍵字
4. Django 後端把條件轉成 SQLite 查詢。
5. 預設回覆三筆最符合條件的活動卡片。
6. 使用者可點「查看更多」取得下一批。

限制：

- 查不到資料時，AI 不得編造。
- 應回覆目前沒有找到符合條件的活動。
- 可提供相近推薦，例如同地區近期活動或其他免費活動。

### 5.4 AI 個人化推薦與互動紀錄

POC 階段：

- 以使用者初次選擇的興趣 Tag 與地區 Tag 作為推薦基礎。

完整系統階段：

- 加入規則式權重推薦。
- 根據互動行為調整推薦權重。

應記錄互動：

- 是否點擊活動卡片
- 是否按下有興趣
- 是否按下沒興趣
- 是否點擊查看更多
- 是否點擊怎麼去
- 是否加入行事曆
- 是否訂閱活動

推薦權重方向：

- 多次點擊某類活動，提高該類 Tag 權重。
- 多次按沒興趣，降低類似活動推薦頻率。

### 5.5 查無資料與防止 AI 幻覺

本系統處理政府活動與公共資訊，AI 回覆必須重視可信度。

規則：

- AI 回覆必須以 SQLite 中已儲存活動資料為依據。
- 查無資料時不得編造。
- 不得自行推測使用者位置。
- 若使用者授權位置，才可依附近地點推薦。

查無資料回覆方向：

```text
目前沒有找到符合條件的活動。
你可以改查：
- 中壢區近期活動
- 桃園市其他免費活動
- 類似市集、戶外、美食類型活動
```

## 6. 與目前資料庫的對應

目前 `Database.events` 已有：

- `Activity`
- `Tag`
- `SourceWebsite`
- `UserProfile`
- `Subscription`
- `ActionLog`

已能支援：

- 活動資料儲存
- 固定 Tag 池
- 使用者偏好 Tag
- 活動訂閱
- 使用者互動紀錄

- `PushCampaign`
- `PushDeliveryLog`
- `ImportRun`
- `ActivityAsset`
- `ActivitySession`
- `ActivityTagSuggestion`
- `AIProcessingLog`
- tracking redirect endpoint

要先支援：

- `PushCampaign`

### 第四優先：AI 標籤審核

新增：

## 8. 重要設計原則

1. AI 不直接創造活動資料。
2. AI 不自由新增 Tag。
3. 所有推薦與回覆必須基於資料庫。
4. 使用者互動要盡量記錄，但不能犧牲正常跳轉體驗。
5. 要跳官方頁又要記錄時，使用中轉 tracking URL。
6. 點擊純互動按鈕時，使用 postback。
7. 後台應保留人工審核能力。
8. 固定 Tag 池是推薦、搜尋與統計的核心。
